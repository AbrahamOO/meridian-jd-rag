# CI/CD Pipeline Standard

## Purpose and Scope

This standard defines how continuous integration and continuous delivery (CI/CD) pipelines are built and secured at Meridian John Doe Financial. The pipeline is the single path through which code becomes a running production artifact, which makes it both the bank's most powerful productivity tool and one of its highest-value attack targets. This standard ensures every pipeline produces signed, attested, reproducible artifacts and that the security gates of the Secure SDLC Policy (MJD-TEC-0001) are enforced by automation rather than by trust.

This standard applies to every pipeline that builds, tests, or deploys software or infrastructure for Meridian J.D., including application pipelines, infrastructure-as-code pipelines (MJD-TEC-0006), and library or package publishing pipelines.

It is binding on all Technology and Platform Engineering teams.

## Definitions

**Pipeline.** The automated sequence of stages (build, test, scan, sign, deploy) triggered by a code change.

**Runner.** The compute that executes pipeline jobs. Runners are ephemeral and isolated.

**Provenance attestation.** A signed statement, generated by the pipeline, that links a built artifact to the exact source commit, builder, and inputs that produced it (per SLSA).

**Artifact signing.** Cryptographically signing a build output so its integrity and origin can be verified before deployment.

**OIDC workload identity.** A short-lived, automatically issued identity the pipeline uses to authenticate to cloud and registry services, replacing long-lived stored credentials.

**Deployment environment.** A named target (development, staging, production) with its own access controls and approval requirements.

**Promotion.** The act of moving a single, immutable artifact from one environment to the next without rebuilding it.

**Waiver.** A time-boxed, attributable approval that allows a specific blocking gate finding to be accepted with a compensating control, recorded against the run.

## 1. Pipeline Integrity Principles

1.1 Pipelines run only from configuration stored in version control alongside the code (pipeline-as-code). No pipeline is configured by hand in a console.

1.2 Pipeline definitions are protected by the same branch protection as application code (MJD-TEC-0009). A change to the pipeline is a code change subject to review.

1.3 Runners are ephemeral: each job runs in a fresh, isolated environment that is destroyed at job end. Self-hosted persistent runners are prohibited for production pipelines because they accumulate state an attacker can poison.

1.4 Third-party pipeline actions and plugins are pinned to an immutable digest, never to a mutable tag. A floating tag reference is a pipeline finding.

1.5 An artifact is built exactly once and then promoted. The same bytes that pass the gates in a lower environment are the bytes that reach production; rebuilding for a higher environment is prohibited because it breaks the chain between tested code and deployed code.

## 2. Pipeline Topology

2.1 Continuous integration (CI) and continuous delivery (CD) are separated into two distinct phases with distinct identities. The CI phase produces and verifies an artifact and holds no production deployment authority. The CD phase consumes a signed, attested artifact and holds deployment authority scoped to a single environment. A failure in CI never reaches CD because no artifact is produced.

2.2 Meridian J.D. operates four standard environments. Each environment has its own access controls, its own pipeline identity, and its own approval requirements. Code and configuration flow strictly from left to right; backward promotion is prohibited.

| Environment | Purpose | Data | Deploy trigger | Approvers required |
|---|---|---|---|---|
| dev | Developer integration and feature validation | Synthetic only | Merge to mainline | Automated, none human |
| test | Automated functional, integration, and security regression | Synthetic only | Successful dev deploy | Automated, none human |
| staging | Production-like rehearsal and performance verification | Masked or synthetic | Successful test deploy | One human approver |
| prod | Live customer-facing service | Live | Approved change record | Two distinct human approvers for high-assurance services |

2.3 Promotion between environments is governed by gates. An artifact may only be promoted into the next environment if it has passed every gate of the prior environment and carries a valid signature and provenance.

| Promotion | Precondition | Gate authority |
|---|---|---|
| dev to test | All CI gates green, artifact signed | Automated |
| test to staging | Functional and security regression green | Automated plus owning team |
| staging to prod | Performance verified, change record approved | Two-person approval (four-eyes) |

2.4 No environment shares a deployment identity with another. The dev pipeline identity cannot authenticate to the production cloud account, and the production deploy identity cannot read dev resources. This per-environment scoping limits the blast radius of any single compromised identity.

## 3. Required Stages and Gates

3.1 Every application pipeline executes the following stages in order. A failing blocking gate stops the pipeline; the artifact cannot proceed.

| Stage | Activity | Gate |
|---|---|---|
| Build | Compile and assemble in an isolated runner | Reproducible build inputs recorded |
| Unit and integration test | Run the test suite | Below coverage baseline blocks |
| SAST | Static analysis | New Critical or High blocks |
| SCA | Dependency and SBOM scan | New Critical or High CVE blocks |
| Secret scan | Scan source and artifact | Any verified secret blocks |
| IaC scan | Policy-as-code on infrastructure | High misconfiguration blocks |
| SBOM generation | Produce CycloneDX SBOM | Missing SBOM blocks |
| Sign and attest | Sign artifact, generate provenance | Unsigned artifact blocks deploy |
| Deploy | Promote to environment | Per section 6 approvals |

3.2 No stage may be skipped silently. A skipped required stage marks the run non-deployable and is recorded as an exception requiring approval per MJD-TEC-0001 section 9.

3.3 Each gate has a defined tool class, a blocking condition, a severity threshold, and a single defined waiver path. Severity is expressed using the CVSS base score where a numeric score applies. No gate is advisory; a gate either blocks or it does not exist.

| Gate | Tool class | Blocking condition | Severity threshold | Waiver path |
|---|---|---|---|---|
| Unit and integration test | Test runner with coverage | Failing test or coverage below baseline | Coverage below 80 percent on changed lines | Owning team lead, recorded against run |
| SAST | Static application security testing | New finding introduced by the change | Critical or High (CVSS 7.0 and above) | Security Architect, time-boxed |
| SCA | Software composition analysis | New vulnerable dependency | Critical or High (CVSS 7.0 and above) | Security Architect, with upgrade plan |
| Secret scan | Verified-secret detection | Any verified live secret | Any confirmed secret, no threshold | Non-waivable, rotate per MJD-TEC-0004 |
| IaC scan | Policy-as-code (MJD-TEC-0006) | High-severity misconfiguration | High and above | Security Architect, time-boxed |
| License and provenance | SBOM policy check | Disallowed license or unresolved provenance | Any prohibited license class | Head of Platform Engineering plus Legal |
| Signature verification | Signature and attestation check | Missing or invalid signature | Any, no threshold | Non-waivable |

3.4 A waiver never silences a gate. A waived finding is still surfaced, still recorded against the run, and still expires on its time box. When the time box lapses the gate blocks again. Critical and High findings that touch authentication, cryptography, or cardholder data flows follow the shortened remediation windows of MJD-SEC-0009 and are not eligible for routine waivers.

## 4. Supply-Chain Controls and Artifact Integrity (SLSA Alignment)

4.1 Every production artifact carries a provenance attestation that meets SLSA build level 3: built on an ephemeral, isolated runner from a verified source, with a non-falsifiable provenance statement signed by the platform.

4.2 Artifacts are signed; the deploy stage verifies the signature and the provenance against the source commit before promotion. An artifact whose provenance does not match its claimed source is rejected.

4.3 The SBOM produced at build time is in CycloneDX format, stored with the artifact, and continuously rescanned for newly disclosed vulnerabilities while the artifact runs in production (MJD-TEC-0001 section 1.6).

4.4 Internal package resolution is configured to prevent dependency-confusion attacks: internal package names resolve only from the internal registry and never from a public registry. A build that attempts to fetch an internally named package from a public source fails the SCA gate.

4.5 Provenance is recorded as an in-toto attestation. Each attestation binds the artifact to a defined set of fields so that any later verifier can reconstruct exactly how the artifact came to exist.

| Provenance field | Meaning |
|---|---|
| subject digest | The content hash of the produced artifact |
| source repository and commit | The exact reviewed commit that was built |
| builder identity | The platform builder that performed the build |
| build entry point | The pipeline definition and invoked workflow |
| build parameters | The inputs and configuration supplied to the build |
| materials | The resolved dependencies and their digests |
| timestamp | When the build started and completed |

4.6 Signature verification happens at admission, not only at build. Before any orchestrator or registry admits an image or artifact into an environment, it verifies the signature and the attestation against the expected source. An artifact that cannot be verified is denied admission, so an artifact built outside the pipeline cannot run even if it reaches the registry.

4.7 Container images are signed with the same platform signing identity that signs other artifacts, and the registry is configured for immutability: a published tag and digest cannot be overwritten. An attacker cannot republish a malicious image under an already-trusted digest, and a rollback always returns to a known, signed image.

## 5. Runner Hardening

5.1 Runners are single-use. Each job receives a freshly provisioned virtual machine or container that exists only for the duration of that job and is destroyed afterward, so no state, cache, or credential persists from one job to the next.

5.2 Runners accept no inbound network connections. They are not addressable from the network and expose no listening service, removing the runner as a remotely reachable target.

5.3 Runner egress is governed by an allowlist. A runner may reach only the source repository, the internal package registry, the artifact registry, the signing service, and the secrets manager. Any other outbound connection is blocked and raises a signal (section 9). An unexpected egress attempt is treated as a possible exfiltration or supply-chain attack.

5.4 Runner base images are pulled by immutable digest from the internal registry and their provenance is verified before use, so a poisoned base image cannot silently enter a build.

5.5 Runners hold no secrets at rest. Any secret a job needs is fetched at job time and lives only in memory for the life of the job. When the runner is destroyed the secret is gone with it.

## 6. Deployment Controls and Strategies

6.1 Deployment to production requires an approved change record per the Change Management and Release Policy (MJD-TEC-0008) and passes the release gate of MJD-TEC-0001 section 1.5.

6.2 Production deployment requires two distinct human approvers for high-assurance services (four-eyes); the approver cannot be the sole author of the change. The two approvers must be different identities, and neither may approve their own change.

6.3 Deployment uses progressive rollout (canary or blue-green) with automated health checks and automated rollback on failure of defined service-level indicators (SLIs).

6.4 Production deployment credentials are never long-lived. The pipeline authenticates to production using OIDC workload identity that is short-lived and scoped to the specific deploy, consistent with the Secrets and Key Management Policy (MJD-TEC-0004).

6.5 The deployment strategy for a service is chosen from the table below according to its assurance tier. Every strategy defines an automated rollback condition so that a degrading deployment is reversed without waiting for a human.

| Strategy | Initial exposure | Bake time | Automated rollback trigger | Suited to |
|---|---|---|---|---|
| Canary | 1 percent, then 5, 25, 50, 100 | 15 minutes per step | Error-rate SLI breach or latency SLI breach | High-assurance customer services |
| Blue-green | Full cutover to standby fleet | 10 minutes on green before retiring blue | Health-check failure on green | Stateful services needing fast reversal |
| Rolling | Replace in batches of 20 percent | 5 minutes per batch | Failed readiness on a batch | Internal and lower-tier services |
| Feature flag | Code deployed dark, exposure decoupled | Variable, controlled by flag | Flag disabled on SLI breach | Risky behavior changes shipped safely |

6.6 Rollback is automated and reversible. When an SLI threshold is breached during a rollout, the pipeline halts the rollout, reverts to the last known-good signed artifact, and raises a signal to the SIEM (MJD-SEC-0009). A human is notified, but reversal does not wait for the human.

## 7. Pipeline Credentials and Secrets

7.1 Pipelines never store long-lived cloud or registry credentials. They use OIDC workload identity federation to obtain short-lived tokens at job time.

7.2 Any secret a pipeline genuinely needs (for example a signing-key grant) is resolved at job time from the secrets manager and is subject to the rotation intervals in MJD-TEC-0004, including the 90-day service-account secret interval for any service-account credential the pipeline holds.

7.3 Pipeline logs are scanned and redacted so a secret accidentally echoed by a build step is masked before the log is stored.

## 8. OIDC Workload Identity

8.1 The pipeline authenticates to cloud, registry, and signing services by exchanging a short-lived, pipeline-issued OIDC token for a scoped, short-lived service credential through federation. No static cloud key, access key, or registry password exists anywhere in the pipeline, in version control, or on a runner.

8.2 Token audience is bound. The federation trust accepts a token only when its audience claim names the specific service being authenticated to. A token minted for one service cannot be replayed against another, which prevents a leaked token from being reused across the estate.

8.3 Federation is scoped per environment. The trust policy for the production cloud account accepts a token only from the production CD identity, on a protected branch, for the named repository and workflow. A token from a dev job, a feature branch, or a fork is rejected at the federation boundary, so a lower environment cannot acquire production authority.

8.4 Tokens are minted at job start and expire shortly after, well within the life of the job. There is no token to steal at rest, and a token captured in transit is useless once the job ends.

8.5 The claims a federation trust evaluates are fixed and minimal. A trust binds on repository, branch or tag, workflow entry point, environment, and audience. Loosening any of these bindings is a high-assurance change reviewed by a Security Architect (MJD-TEC-0001 section 5).

## 9. Isolation and Least Privilege

9.1 The build environment for untrusted contributions (for example, external pull requests) never has access to production credentials, signing keys, or the ability to publish artifacts. It runs in a strictly read-only, no-secrets context.

9.2 Each pipeline's identity is scoped to only the environments and resources it must touch. A development pipeline cannot deploy to production.

9.3 A pipeline identity holds the minimum permission set for its phase. A CI identity can read source and write artifacts but cannot deploy; a CD identity can deploy to one environment but cannot rewrite source or signing configuration.

## 10. Pipeline-as-Code Governance

10.1 Every pipeline definition lives in version control under the same branch protection as application code (MJD-TEC-0009). A change to a pipeline is a reviewed code change, not a console edit, and is itself subject to the gates of this standard.

10.2 Reusable, hardened pipeline templates are maintained centrally by Platform Engineering. Teams compose pipelines from these templates rather than authoring security-relevant logic from scratch, so the signing, attestation, and gate-enforcement steps are inherited and cannot be quietly removed by an individual team.

10.3 A change that weakens a centrally provided template (removing a gate, loosening a digest pin, broadening a federation trust) is a high-assurance change reviewed by a Security Architect and cannot be self-approved by the proposing team.

10.4 Drift between a team pipeline and the central template is detected automatically. A pipeline running on an outdated or modified template is flagged for the owning team and the Pipeline Platform team.

## 11. Auditability

11.1 Every pipeline run produces an immutable record: who triggered it, the source commit, the gates that passed, the artifact signature, and the provenance. This record is retained at least 13 months (MJD-CMP-0008) and feeds the audit trail standard (MJD-FIN-0005 for change evidence and MJD-SEC-0009 for security events).

11.2 The pipeline run record links to the change record (MJD-TEC-0008) for any production deployment, so an examiner can trace a running production artifact backwards through deploy approval, gate evidence, the signed artifact, its provenance, and the reviewed source commit, with no gap in the chain. This end-to-end traceability is the evidence the bank presents to demonstrate that PCI DSS 4.0 Requirement 6.5 change-management controls operate effectively.

11.3 Gate overrides and exceptions, if any are approved under MJD-TEC-0001 section 9, are recorded against the run with the approver identity and justification. An approved exception is never silent; it is a visible, attributable event in the audit record.

## 12. Delivery and Security Metrics

12.1 The Pipeline Platform team publishes delivery and security health metrics each cycle. Delivery throughput and stability are tracked using the four DORA measures, and pipeline security posture is tracked alongside them so that speed is never traded against assurance.

| Metric | Definition | Direction |
|---|---|---|
| Deployment frequency | How often the pipeline deploys to production | Higher is healthier |
| Lead time for change | Time from commit merged to running in production | Lower is healthier |
| Change-failure rate | Share of deployments causing a degradation or rollback | Lower is healthier |
| Mean time to restore (MTTR) | Time to recover service after a failed change | Lower is healthier |
| Security gate pass rate | Share of runs passing all security gates without a waiver | Higher is healthier |
| Active waiver count | Open, time-boxed gate waivers | Lower is healthier |

12.2 A declining security gate pass rate or a rising active waiver count is reviewed by a Security Architect, because it signals that controls are being routed around rather than satisfied. Metrics are evidence, not vanity numbers, and feed the periodic review of this standard.

## 13. Worked Example: A Commit From Pull Request to Production

13.1 The following walks a single change through every gate to show how the controls compose in practice for a high-assurance service.

13.2 An engineer opens a pull request. Branch protection (MJD-TEC-0009) requires review; a second engineer reviews and approves. The untrusted-contribution rules do not apply because this is an internal branch, but the build still runs with no production credentials.

13.3 On merge to mainline, the CI phase starts on a freshly provisioned ephemeral runner with no inbound access and an egress allowlist. The runner pulls its base image by immutable digest and resolves dependencies, with internal package names resolving only from the internal registry.

13.4 The build compiles the artifact once and records reproducible build inputs. The test gate runs and meets the coverage baseline. SAST finds no new Critical or High. SCA finds no new Critical or High CVE. The secret scan finds no verified secret. The IaC scan passes policy. A CycloneDX SBOM is generated and stored with the artifact.

13.5 The sign-and-attest stage signs the artifact using a signing key held in the KMS and never exported, and generates an in-toto provenance attestation meeting SLSA build level 3, binding the artifact to the reviewed commit, the builder, and the resolved materials.

13.6 The artifact is promoted (not rebuilt) to dev, then test, then staging. At each admission the orchestrator verifies the signature and provenance before the artifact runs. Staging promotion requires one human approver and a passing performance check.

13.7 For production, a change record is approved (MJD-TEC-0008) and two distinct human approvers sign off (four-eyes); neither is the sole author. The CD phase authenticates to the production account using a short-lived OIDC token whose audience and environment claims bind it to production, scoped to this deploy only.

13.8 The rollout is canary: 1 percent, then 5, 25, 50, and 100, with a 15-minute bake at each step against error-rate and latency SLIs. Had an SLI breached, the pipeline would have halted, rolled back automatically to the last known-good signed artifact, and raised a signal to the SIEM (MJD-SEC-0009). The rollout completes cleanly.

13.9 The run leaves an immutable record retained at least 13 months (MJD-CMP-0008) that an examiner can trace backward from the running artifact, through the deploy approvals and gate evidence, to the signed artifact, its provenance, and the reviewed commit, with no gap.

## 14. Pipeline Resilience and Integrity Monitoring

14.1 The pipeline platform is itself a high-assurance system. Changes to runner images, the signing service, and the OIDC federation configuration are high-assurance changes (MJD-TEC-0001 section 5) and are reviewed by a Security Architect.

14.2 The signing service's keys are held in the KMS and never exported, consistent with the Secrets and Key Management Policy (MJD-TEC-0004 section 2.2). A compromise of the build platform must not yield a usable signing key.

14.3 The integrity of runner base images is verified before use: images are pulled by immutable digest from the internal registry and their provenance is checked, preventing a poisoned base image from silently entering builds.

14.4 Anomalous pipeline behavior (an unexpected network egress from a build, a job attempting to read a secret it does not need, a deploy attempted outside an approved window) raises a security signal to the SIEM (MJD-SEC-0009) for investigation.

## 15. Roles and Responsibilities

**Software Engineer.** Defines pipeline-as-code from the central hardened templates, keeps action pins on immutable digests, fixes blocking gate failures at the source rather than seeking waivers, never circumvents the pipeline to deploy by hand, and ensures the artifact built once is the artifact promoted. The Software Engineer is responsible for keeping the change traceable from commit to deploy.

**Security Architect.** Owns the pipeline threat model and the gate thresholds, sets and reviews the CVSS severity bars and the waiver paths, reviews changes to signing, provenance, and OIDC federation, approves any time-boxed pipeline exception, and reviews declining security gate metrics. The Security Architect is the approval authority for any high-assurance change to the delivery path.

**Platform Engineering (Pipeline Platform team).** Operates the runner fleet, the signing and attestation service, the artifact registry, and the OIDC federation; maintains the central hardened templates and detects drift; ensures runners are ephemeral, have no inbound access, and enforce egress allowlists; and verifies runner base image provenance.

**Service Owner.** Selects the deployment strategy for the service from the approved table, defines its SLI rollback thresholds, and nominates the human approvers for production deploys, ensuring the four-eyes requirement is met for high-assurance services.

**Head of Platform Engineering.** Owner of this standard, accountable for the integrity of the delivery path and for the publication of delivery and security metrics.

## 16. Exceptions and Escalation

16.1 No exception is granted to: artifact signing and provenance verification, the prohibition on persistent production runners, the use of long-lived production credentials, the isolation of untrusted-contribution builds from secrets, or the binding of OIDC tokens to a specific audience and environment. These are non-waivable.

16.2 Other deviations require an exception approved by a Security Architect and the Head of Platform Engineering, time-boxed with a compensating control and recorded against the affected runs.

16.3 A deployment that bypassed the pipeline, an unsigned artifact reaching production, a compromised runner, or a federation trust that accepted a token from an unauthorized source is a security incident escalated under the Incident Response Plan (MJD-SEC-0006).
